# Tactical Analyst Agent

An agentic pipeline that turns raw football match event data into a tactical analysis report — and checks its own numbers before showing them to you.

Built to answer a specific question: **can an LLM write a trustworthy report grounded in real stats, and how would you know if it didn't?**

```
fetch_data → compute_stats → write_narrative → evaluate → (retry once on failure) → done
```

## Why this exists

Most "agent demo" projects are a single prompt with a nice UI. This one is built around the part that actually matters in production: an **evaluation layer that catches hallucinated numbers before they ship**. Every number the model writes is checked against the source stats it was given — deterministically, without another LLM call, so the check itself doesn't need trusting.

## What it does

1. **Fetches real data** — free, public match event data from [StatsBomb Open Data](https://github.com/statsbomb/open-data) (no scraping, no API key needed for this step). No mock data anywhere in this pipeline.
2. **Computes stats** — possession, shot quality (a transparent, hand-built xG proxy — not a black box), pass accuracy, territory, and pressing intensity, all traceable back to formulas in `stats_engine.py`.
3. **Writes a narrative** — Claude turns the structured stats into a tactical report, explicitly instructed to only use numbers it's been given.
4. **Evaluates the report** — `eval_harness.py` scans the generated text for every number, checks it against the source stats (with sensible rounding tolerance), and flags hallucinations, missing coverage, or team mixups. If it fails, the pipeline retries the narrative step once.
5. **Deploys as a live demo** — a Streamlit app where you pick a match and watch the pipeline run.

## Example output

*(2022 World Cup Final, Argentina vs France — generated by the pipeline, eval-passed)*

> Argentina's tactical plan produced a curious asymmetry: they had less of the ball but generated far more danger with it.
>
> Despite holding just 46.3% possession to France's 53.7%, Argentina created 6.64 expected goals from 24 shots, more than 50% higher than France's 4.0 xG from 14 attempts. That gap reflects a team happy to cede sterile possession in deep areas while striking with intent once inside the final third, where they registered 215 entries against France's 157.
>
> France's threat was more compressed: fewer shots, but a similar average shot quality (0.286 xG/shot vs Argentina's 0.277), suggesting their chances came from moments of individual quality rather than sustained territorial control. Their 45 high-press actions in Argentina's defensive third — more than Argentina's own 31 — point to a team trying to disrupt build-up rather than dominate it directly.
>
> **Verdict:** Argentina controlled the game's most dangerous moments without controlling the ball itself — a deliberate trade-off that nearly proportionally matches the final scoreline.

*(Note: the match brief includes StatsBomb's full event stream, which for this match covers extra time and the penalty shootout — goal counts above reflect that.)*

## Architecture

```
src/
  data_loader.py      # StatsBomb open-data client
  stats_engine.py      # raw events -> structured MatchBrief (transparent formulas)
  narrative_agent.py   # MatchBrief -> LLM-generated report
  eval_harness.py      # report + MatchBrief -> pass/fail + flagged issues
  agent_graph.py        # LangGraph orchestration wiring the above into a pipeline
app.py                  # Streamlit UI
tests/                  # deterministic unit tests (no API key needed)
```

**Why LangGraph and not a single prompt:** each stage is independently testable and swappable. You can unit test the stats engine and eval harness with zero API calls (see `tests/`), and the narrative step can be swapped for a different model or prompt without touching anything upstream.

**Why a hand-built xG formula instead of a trained model:** it keeps the whole pipeline auditable end-to-end — every number in the final report traces back to a documented formula in `stats_engine.py`, not a model with hidden weights.

## Running it

```bash
git clone <this-repo>
cd tactical-agent
pip install -r requirements.txt

# Run tests (no API key needed)
pytest tests/ -v

# Run the full pipeline from the command line
export ANTHROPIC_API_KEY=your_key_here
python src/agent_graph.py

# Or launch the interactive app
streamlit run app.py
```

## Deploying the live demo

Free options, no AWS bill required:

- **[Streamlit Community Cloud](https://share.streamlit.io)** — connect this repo, set `app.py` as the entrypoint, add `ANTHROPIC_API_KEY` as a secret. Live in ~2 minutes.
- Alternative: containerize with the included pattern and deploy to AWS Lambda/Fargate behind API Gateway if you want to demonstrate the same serverless deployment pattern as production ML systems.

## Limitations (documented on purpose)

- The xG proxy uses distance/angle only — it doesn't know a shot is a penalty, so it underrates penalties specifically (see `tests/test_stats_engine.py` for the documented gap vs. real-world penalty conversion rates). A real model would add a shot-type feature; this one is intentionally simple so it stays fully explainable.
- The eval harness catches *numeric* hallucinations, not qualitative misrepresentation (e.g. correctly-cited numbers used to support a wrong tactical claim). That's a harder problem and out of scope here — worth flagging in an interview as the natural next step.
- Currently scoped to StatsBomb's open competitions (World Cups, a handful of leagues) since that's what's freely available without a data license.
